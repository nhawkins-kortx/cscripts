cscript amend-stack

system interaction

How the command interacts with your repository: it amends the current commit with the staged index, then replays the whole dependent-branch forest onto the rewritten commit - driving rebases in the current worktree, in linked worktrees, and finally the remote. The load-bearing rule is the ordering: nothing is rewritten until every guard passes and you confirm.

  • Discover
  • Capture old tips
  • Preflight guards
  • Plan + confirm
  • Amend + restack
  • Push

What it drives

One process orchestrates four git surfaces. All worktrees share a single object store, which is why a branch checked out elsewhere can be rebased in place once the new base commits exist.

Ordering invariant

Why the flow is shaped this way:

discover → capture tips →
preflight → plan → confirm →
amend + rebase → push

Every abort condition is evaluated before the amend, so a run can never end up amended-but-stuck.

Preflight guards

  • 🔵 detached / unborn HEAD
  • 🔵 current worktree mid-rebase/merge
  • 🔵 unstaged tracked changes (index-only tool)
  • 🔵 merge commit in a selected range
  • 🔵 selected branch busy in another worktree

Skipped under --dry-run so the plan is always previewable.

Rebase cascade

Each dependent, parents first:

git rebase --autostash
  --onto <parent> <oldParentTip> <branch>

<parent> is a branch name (resolves to its already-rebased tip); <oldParentTip> is a SHA captured before any mutation. That pairing is what lets fan-out trees replay cleanly.

Verified by 71 integration tests across four adversarial review rounds. Documented design edges: empty/EOF input follows the [Y]/n default; merge stacks are refused, not rebased; message edits are single-line.
